What is Antenna?

  • Open-source platform from the Mila Institute
  • Processes images from automated wildlife camera traps in the field
  • Classifies insect species using deep learning — supporting biodiversity monitoring at scale
  • Used by multiple research institutions across different deployment environments

insectai.org — open source, community-driven

Caption: Antenna web platform — researchers submit image batches and review classification results.

The Scaling Problem

As adoption grew, jobs went from hundreds to millions of images. Two interconnected problems emerged:

  Integration barrier

Third-party researchers couldn't easily integrate their own ML pipelines — requiring a public HTTP endpoint was a deployment blocker

  Throughput ceiling

Platform couldn't scale processing to large jobs or multiple models — synchronous design limited throughput and flexibility

This talk: two engineering changes that solve both — async pull architecture & PyTorch GPU optimization

Part 1: Async Pull Architecture

Inverting the communication model to unlock distributed ML inference

How the Original System Worked

  • Antenna pushed image batches to external ML processing services (PS) over HTTP
  • Each PS had to expose a public HTTP endpoint — hard in university HPC environments or behind firewalls
  • Synchronous: Antenna waited for each response before moving on
  • Scalability was entirely the PS implementor's problem

Diagram A — Push Model (before)

The Redesign — Pull Model

  • Invert the direction: PS workers poll Antenna's job queue and pull task batches on their own schedule
  • PS no longer needs a public endpoint — it initiates all outbound connections
  • Workers run behind a firewall, on an HPC cluster, or on a local machine
  • Multiple workers poll the same queue independently — horizontal scaling emerges naturally

Diagram B — Pull Model (after)

Worker Implementation — ami worker

  • Ships as ami worker CLI in the ami-data-companion package
  • Configuration via environment variables or a .env file (API URL, auth token, batch size)
  • Worker loop: poll /jobs → fetch batch → run ML pipeline → POST results
  • Any number of workers can run in parallel against the same queue

Install: pip install ami-data-companion

3-step startup 

export AMI_API_URL=https://your-antenna.org
export AMI_API_KEY=your-token
export AMI_BATCH_SIZE=32

ami worker register --pipeline insects-v2
ami worker
Source: github.com/RolnickLab/ami-data-companion

Part 1 Results

  Deploy anywhere

Workers run anywhere with outbound HTTP — HPC clusters, university firewalls, local machines. No public endpoint required.

  Horizontal scaling

Add more workers without changing Antenna. Each worker independently polls and claims batches from the same queue.

  Extensibility

New ML pipelines register and pull independently. Researchers build custom workers for their own models.

  Multi-institution

Workers run across research institutions on different hardware configurations — no central coordination needed.

Key insight: inverting push → pull makes distributed ML inference self-scaling — each new worker adds capacity automatically.

Part 2: PyTorch GPU Inference Optimization

From 1,674 seconds to 108 seconds — a 15× improvement on a single GPU

Why Batch Data Loading Matters

  Request / response or push

  • Caller sends images, then waits or guesses when to send more
  • Caller cannot see worker throughput, GPU memory, or safe concurrency
  • Network and image I/O stay on the critical path

  Batch data loader

  • Worker is in the driver seat: it can choose batch size and pull when ready
  • Multiple workers load and decode images in the background
  • Prefetched batches keep data ready while the GPU processes the current batch

Design shift: image loading becomes a queue the worker keeps full, instead of a blocking request the pipeline waits on.

  • GPU utilization was poor — significant idle gaps visible in profiling
  • Root causes: CPU–GPU transfer bottlenecks, synchronous result posting, suboptimal data loading

Diagram C — GPU Utilization Before Optimization

Fix 1 — Keep Data on the GPU Between Stages

  • Detection and classification are two sequential model passes
  • Original code converted intermediate tensors back to PIL images between passes
  • Forced a GPU → CPU → GPU round-trip for every image in the batch
  • Fix: keep tensors on GPU and pass them directly from detector to classifier

Diagram D — Data Flow Before / After

Fix 2 — Pinned Memory & Batch Collation

  • pin_memory=True enables DMA transfers — CPU memory is page-locked, avoiding an extra copy on the way to the GPU
  • Batch collation (stacking images into a tensor) was running in the main process, blocking the GPU pipeline for large images
  • Fix: move collation into DataLoader workers so it runs in parallel with GPU computation

DataLoader configuration

DataLoader(
  dataset,
  batch_size=32,
  pin_memory=True,      # DMA transfer
  num_workers=4,        # parallel collation
  collate_fn=collate,   # off main thread
)

Effect: batch prep runs on CPU workers while the GPU processes the previous batch — overlap, not sequential.

Fix 3 — CUDA Streams for Prefetching

  • GPU was idle waiting for the next batch to transfer from CPU
  • CUDA streams allow concurrent execution: stream A processes batch N while stream B prefetches batch N+1 into GPU memory
  • Result: near-continuous GPU utilization instead of alternating compute/transfer cycles

Diagram E — CUDA Stream Pipeline

Fix 4 — Async Result Posting

  • After each batch, results were posted to Antenna synchronously — GPU sat idle during the HTTP round-trip
  • Fix: ResultPoster runs in a background thread
  • Worker loads and processes batch N+1 while batch N results are being posted

Same principle as CUDA streams applied at the API layer — overlap I/O with compute at every level of the stack.

Conclusion & Takeaways

Generalizable patterns for federated ML inference

Summary & Generalization

  Part 1 takeaway

Inverting push → pull makes distributed ML inference dramatically easier to deploy and scale. Workers run anywhere with outbound HTTP.

  Part 2 takeaway

Four concrete PyTorch optimizations — each addressing a different bottleneck — compound to a 15× gain. The pattern generalizes to any GPU-accelerated workload.

  Both contributions generalize

Applicable to any image classification workload — not just entomology. The async pull architecture works for any ML inference service that can poll a job queue. The GPU optimizations apply to any multi-stage deep learning pipeline.
